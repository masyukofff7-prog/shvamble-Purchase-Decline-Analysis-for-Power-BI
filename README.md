 shvamble Purchase Decline Analysis for Power BI

 О проекте

Этот проект посвящён анализу снижения объёмов закупок у поставщика shvamble в 2024–2025 годах.  
Цель анализа - понять, по каким товарам закупки снизились, продолжают ли эти товары закупаться у shvamble или были переведены на других поставщиков, а также показать возможные причины переключения поставок.

Проект ориентирован на построение аналитической модели в Power BI с предварительной подготовкой данных через MySQL и Power Query.

Основа проекта - бизнес-постановка и технический дизайн из файла  

---

 Бизнес-задача

Нужно было ответить на два ключевых вопроса:

1. Почему в 2024–2025 годах снизились объёмы закупок у shvamble.
2. Остались ли эти товары у shvamble или были переведены на других поставщиков.

Под “текущим поставщиком” понимается поставщик, который в таблице `product_supplier` отмечен как `main = 1`.

---

 Что должно было получиться в отчёте

В итоговую таблицу в Power BI должны были войти:

- исходные поля из файла поставщика;
- внутренние идентификаторы товара: `supplier_num`, `ext_id`, `product_id`;
- данные текущего основного поставщика:
  `supplier_id`, `name`, `price`, `discount_price`, `discount_qty`;
- условия самого shvamble для сравнения по той же SKU, чтобы можно было показать возможную причину переключения на другого поставщика.

---

 Источники данных

В проекте использовались следующие источники:

- файл поставщика с артикулами и количествами по годам;
- таблица `product_supplier`;
- таблица `products`;
- таблица `supplier`.

---

 Логика сопоставления данных

Основная сложность заключалась в том, что поставщик использует свой идентификатор товара - Artikelnummer / supplier_num,  
а внутри системы используются `product_id` и `ext_id`.

Логика сопоставления была следующей:

1. Артикул из файла поставщика связывается с `product_supplier.supplier_num` для `supplier_id = 1` (shvamble).
2. Через `product_supplier` определяется `product_id`.
3. Через таблицу `products` находится `ext_id`.
4. По `product_id` определяется текущий основной поставщик, где `main = 1`.
5. Через таблицу `supplier` подтягивается название текущего поставщика. 

---

 Что я сделал

 1. Спроектировал слой подготовки данных в SQL

Чтобы минимизировать нагрузку на Power BI и упростить модель, основная логика объединения была вынесена в MySQL:

- отбор shvamble как поставщика с `supplier_id = 1`;
- выделение текущего основного поставщика через `main = 1`;
- объединение с карточкой товара;
- получение ценовых условий shvamble и текущего поставщика в одной выборке;
- расчёт диагностического признака `is__main`. 

 2. Подготовил структуру enrichment-датасета

SQL-слой формировался как обогащающий набор данных, ключом в котором выступал `supplier_num`.  
Это позволяло затем удобно делать один merge в Power Query с исходным supplier-файлом.

 3. Продумал обработку данных в Power Query

Для загрузки и очистки данных предполагались следующие шаги:

- поднятие заголовков;
- переименование колонок в стабильные технические имена;
- удаление итоговой строки `Gesamtmenge`;
- приведение типов данных;
- `Left Outer Join` между таблицей поставщика и SQL-enrichment-слоем;
- раскрытие нужных полей после merge.

 4. Продумал аналитическую модель в Power BI

Для объяснения снижения объёмов были предусмотрены меры:

- объём за 2024;
- объём за 2025;
- абсолютное изменение;
- процентное изменение;
- количество SKU;
- количество SKU, где shvamble больше не является основным поставщиком;
- потенциальное ценовое преимущество текущего поставщика;
- оценка экономии на объёме 2025 года.

 5. Добавил контроль качества данных

Так как проект связан со сверкой разных идентификаторов, отдельно были предусмотрены проверки на:

- несопоставленные артикулы;
- отсутствие основного поставщика;
- дубли сопоставления одного `supplier_num` на несколько `product_id`.

 Ключевой результат

В результате была спроектирована модель, которая позволяет:

- связать supplier-данные с внутренней товарной системой;
- определить текущего поставщика по каждому SKU;
- сравнить коммерческие условия shvamble и текущего поставщика;
- объяснить снижение закупок не только через объёмы, но и через возможную ценовую причину;
- выявить проблемы качества мастер-данных, влияющие на интерпретацию результата.
